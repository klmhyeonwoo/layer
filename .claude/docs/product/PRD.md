# PRD: Layer 회고 협업 플랫폼

## 개요
Layer는 팀과 개인이 스페이스 단위로 회고를 만들고, 정해진 템플릿에 따라 회고를 작성하며, 작성 결과를 분석해 인사이트와 액션 아이템으로 이어갈 수 있게 하는 회고 협업 플랫폼이다. 모바일과 데스크탑 환경을 모두 지원해 회고 생성, 참여, 작성, 분석, 후속 실행 관리를 하나의 흐름으로 제공한다.

## 문제 정의
팀 회고는 템플릿 선정, 참여자 관리, 작성 독려, 결과 정리, 후속 액션 관리가 분산되기 쉽다. 이로 인해 회고가 일회성 기록에 머물거나, 실행 가능한 개선 과제로 연결되지 못한다. Layer는 회고 전후 과정을 제품 안에서 연결해 팀이 반복 가능한 회고 루틴을 만들도록 돕는다.

## 목표
- 사용자는 스페이스를 만들고 팀원을 초대해 회고를 함께 진행할 수 있다.
- 사용자는 기본 템플릿, 추천 템플릿, 커스텀 템플릿을 활용해 회고를 생성할 수 있다.
- 사용자는 회고 질문에 답변하고 임시 저장 또는 제출 흐름을 통해 회고를 완료할 수 있다.
- 사용자는 회고 결과 분석과 액션 아이템을 통해 다음 실행 과제를 확인할 수 있다.
- 사용자는 모바일과 데스크탑에서 주요 회고 흐름을 일관되게 사용할 수 있다.

## 비목표 (Out of Scope)
- 백엔드 API 서버 자체 구현
- 조직 단위 과금, 결제, 플랜 관리
- 실시간 공동 편집
- 외부 업무 도구와의 양방향 동기화
- 관리자용 대시보드 전체 구현

## 타깃 사용자
- 팀 회고를 주기적으로 운영하는 팀 리더 또는 스크럼 마스터
- 회고에 참여해 의견을 작성하고 결과를 확인하는 팀원
- 개인 또는 팀 목표를 액션 아이템으로 관리하려는 사용자
- 모바일에서 빠르게 회고에 참여하고, 데스크탑에서 회고 현황을 넓은 화면으로 관리하려는 사용자

## 주요 기능

### 인증 및 사용자 프로필
- 사용자는 소셜 로그인을 통해 서비스에 진입할 수 있다.
- 사용자는 최초 로그인 후 닉네임을 설정할 수 있다.
- 사용자는 내 정보 화면에서 프로필을 수정하고 계정을 탈퇴할 수 있다.

### 스페이스 관리
- 사용자는 스페이스를 생성하고 이름, 분야, 이미지 등 기본 정보를 설정할 수 있다.
- 사용자는 초대 링크를 통해 스페이스에 참여할 수 있다.
- 사용자는 스페이스 정보를 수정하거나 나가기, 삭제 등의 관리를 수행할 수 있다.
- 리더는 멤버 목록을 확인하고 리더 변경 또는 멤버 내보내기를 할 수 있다.

### 회고 생성
- 사용자는 스페이스 안에서 새 회고를 생성할 수 있다.
- 사용자는 회고 제목, 기간, 마감일, 템플릿을 선택할 수 있다.
- 사용자는 추천 템플릿 탐색 또는 검색을 통해 목적에 맞는 템플릿을 고를 수 있다.
- 사용자는 커스텀 질문을 편집해 팀에 맞는 회고 템플릿을 구성할 수 있다.

### 회고 작성
- 사용자는 회고 질문 목록을 확인하고 답변을 작성할 수 있다.
- 사용자는 작성 중인 답변을 임시 저장할 수 있다.
- 사용자는 회고 작성 완료 후 완료 화면에서 다음 행동으로 이동할 수 있다.

### 분석 및 인사이트
- 사용자는 회고 분석 화면에서 팀 또는 개인의 회고 결과를 확인할 수 있다.
- 사용자는 만족도, 인사이트, 작성 답변 기반 분석 결과를 확인할 수 있다.
- 사용자는 회고 결과에서 도출된 후속 실행 과제를 확인할 수 있다.

### 액션 아이템
- 사용자는 개인 또는 팀 액션 아이템 목록을 확인할 수 있다.
- 사용자는 액션 아이템을 추가, 수정, 삭제할 수 있다.
- 사용자는 홈 또는 목표 화면에서 후속 과제 진행 상황을 확인할 수 있다.

### 공지 및 지원 정보
- 사용자는 공지사항, 도움말, 라이선스, 약관, 개인정보 처리방침을 확인할 수 있다.
- 사용자는 피드백을 전달할 수 있다.

## 성공 지표
| 지표 | 현재 | 목표 |
|------|------|------|
| 스페이스 생성 완료율 | 측정 필요 | 생성 시작 사용자 중 70% 이상 |
| 회고 생성 완료율 | 측정 필요 | 생성 시작 사용자 중 70% 이상 |
| 회고 작성 완료율 | 측정 필요 | 작성 시작 사용자 중 80% 이상 |
| 액션 아이템 생성률 | 측정 필요 | 완료된 회고 중 50% 이상 |
| 모바일/데스크탑 주요 플로우 오류율 | 측정 필요 | 주요 플로우별 1% 미만 |

## 제약사항
- 프론트엔드는 API 서버 계약에 의존하므로 API 변경 시 타입과 훅을 함께 갱신해야 한다.
- 모바일과 데스크탑 라우트가 같은 코드베이스 안에 공존하므로 기능 변경 시 양쪽 화면 영향을 확인해야 한다.
- 인증 토큰, 리프레시 토큰, WebView Bridge, PWA 업데이트 등 런타임 환경 차이를 고려해야 한다.
- 보호 브랜치 정책상 `feature/*`, `fix/*`, `chore/*` 브랜치에서 PR 기반으로 작업해야 한다.
